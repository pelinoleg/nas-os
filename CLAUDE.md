# NAS-OS — инструкции проекта для Claude Code

Домашняя NAS-система для Raspberry Pi 5: движок настройки + веб-панель в стиле «десктоп-окружения».
Этот файл переживает переустановку (он в git), поэтому `claude`, запущенный в папке репозитория,
сразу знает архитектуру и правила. Память в `~/.claude/` — локальная и при переустановке пропадает;
источник правды по проекту — ЭТОТ файл + история git.

## Что это и где что лежит
Репозиторий: **github.com/pelinoleg/nas-os** (ветка `main`). `gh` авторизован.
- `nas-web.py` (~5000 строк) — **stdlib-only** HTTP-сервер (ThreadingHTTPServer) от **root**, порт 80.
  Отдаёт статику из `web/`, REST `/api/*`, WebSocket `/ws/term`. Мост к движку:
  `bash nas-wizard.sh api <action>` через env `NASW_*`. Никаких внешних Python-зависимостей.
- `nas-wizard.sh` (~2500 строк) — движок+инсталлятор. Интерактивный (whiptail) и headless
  API-режим (`nas-wizard.sh api <action>`, вызывается из nas-web без whiptail).
- `web/desktop.html` — весь десктоп: инлайн `<style>` + инлайн `<script>` (ванильный JS, без сборки).
- `web/setup.html` — мастер первичной настройки (отдельная страница).
- `web/sw.js` — service worker (network-first; HTML — только из сети).
- `install.sh` — `curl -fsSL .../install.sh | sudo bash` → `git clone/reset --hard origin/main`
  в `/opt/nas-os`, ставит службу `nas-web` оттуда. **Переустановка воспроизводит текущий код 1:1.**
  Запускает `nas-wizard.sh api system` = `stage_system_apply` (авто-база, headless). Что база
  ставит из коробки (2026-07-20 расширена): пакеты (STACK/UTIL/PI + docker + gh), watchdog,
  uas-off, usb-timeout, smartd-guard, thumbs-таймер, **avahi+libnss-mdns** (`.local`),
  **Samba** (`install_smb_shares` — вкладка Sharing живая, шар ноль), **drivetemp** (темп. SATA),
  notify-helper, motd, **netguard** (сетевая устойчивость — теперь в базе, не в стадии backup)
и **unattended-upgrades** (2026-07-20: авто-security-обновления в базе — бокс без присмотра не
копит CVE; канал только security, тумблер в панели остаётся). Целевая ОС — **Debian/RaspiOS**:
install.sh на другой ОС (Ubuntu) громко предупреждает и спрашивает подтверждение через `/dev/tty`
(пайп `curl|bash` — stdin занят скриптом), headless-обход `NASOS_FORCE_OS=1`. Пакеты, которых
установщик не нашёл в репах (переименование в новом Debian), собираются в
`/var/lib/nas-wizard/skipped-packages` (`report_skipped_packages`, громкая сводка в конце
`api system`; «Pi packages» на не-Pi — ожидаемо, не шумим) → `stats().missing_pkgs`
(dpkg-перепроверка, TTL 5 мин) → несмахиваемая плашка `.basepkgs` внизу стола
(`paintBasePkgs`): исчезает САМА, когда пакеты доставлены, кнопки «скрыть» нет намеренно.
  Дальше — диски/пул/шары/стеки/тюнинг в панели/веб-мастере (железо у всех разное).

## Рабочий цикл разработки
1. Правишь файл.
2. **Питон**: `sudo systemctl restart nas-web` (обязательно — код в памяти службы).
   **Статика** (`web/*`): рестарт не нужен, сервер читает с диска; но браузер/кэш — см. ниже.
3. Проверка: `python3 -m py_compile nas-web.py`, `bash -n nas-wizard.sh`, для JS —
   вынуть `<script>` и `node --check`, для CSS — проверить баланс `{` / `}`.
4. **Всегда коммить И пушить** каждую законченную правку (пользователь может уйти/выключить NAS).
   Подпись коммита: `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>`.
5. **CLAUDE.md — часть правки, не после.** Любая новая фича/грабля/изменение состояния —
   сразу дописывается в CLAUDE.md (секция «Состояние» и/или «Ключевые грабли») в ТОМ ЖЕ коммите
   или следом, и пушится. Файл не должен отставать от кода: он единственный источник правды,
   переживающий переустановку. Правило пользователя (2026-07-22): держать CLAUDE.md всегда
   актуальным и всё пушить — по умолчанию, без напоминаний.
6. Дев-хост = сам NAS (Pi 5). Служба `nas-web` — `enabled` (стартует после ребута сама).

## ЖЕЛЕЗНЫЕ ПРАВИЛА (нарушение = сломанная переустановка / несогласованный UI)

### 1. Полнота установщика (wizard-completeness)
Любое ГЛОБАЛЬНОЕ изменение системы (udev-правило, systemd-юнит, пакет, конфиг в `/etc`,
sudoers, fstab-логика) ОБЯЗАНО быть прописано в `nas-wizard.sh`/`install.sh`, иначе после
переустановки оно исчезнет. Если правишь «живьём» на боксе — параллельно правь генератор
в визарде. Секреты/конфиги пользователя (пароли, `nas-backup.json`, `desktop.json`, задачи)
в репо НЕ входят — они восстанавливаются из «Настройки → Бэкап настроек» (`_bk_sources()`).
**Что кладёт бэкап настроек**: весь `~/nas-config/` (тема/окна/ярлыки/обои, glance.json,
screen.json, storage.json — выбор основного хранилища, профили и история приложения Backup,
кэш иконок), `etc/nas-os/{webauth,nas-backup}.json` (пароль панели, главный NAS), `etc/nas-wizard/`
(motd, usb-import), Samba (`smb.conf` + `passdb.tdb` — пароли SMB-юзеров), `opt/stacks/*/`
(compose+.env стеков), SSH-ключ push-бэкапов (`/root/.ssh/nas-backup`). `reference/` — только
справочно (НЕ авто-восстановление): `fstab`/`snapraid.conf`/`exports` (диски другие — руками) и
`netplan/*.yaml` (Wi-Fi-пароль/статический IP; секретная секция «Сеть»). Секретные секции
(webauth/nasbackup/network) прячутся за отдельным подтверждением при восстановлении. Дефолтный
путь — основное хранилище (пул или выбранный USB через `storage_root()`), без него — системный
`/var/backups/nas-os` (эфемерно, панель предупреждает).

### 2. Дизайн-система (весь новый UI обязан следовать)
Токены — CSS-переменные, тема через `:root[data-theme]` + `@media prefers-color-scheme`.
- НЕ использовать солид-цвета (`#hex`, `var(--bg)`, `var(--surface)`) как заливку карточек/панелей
  ВНУТРИ окон — только стекло `--glass`/`--card`. `var(--bg)` (сплошной) = ТОЛЬКО рабочий стол/body.
- **Elevation в тёмной теме = белый слой с прозрачностью** поверх материала (не осветление того же цвета).
- **Активные табы/сегменты/сайдбары — «белая фишка»**: `background:var(--sel)` + акцентный ТЕКСТ +
  `box-shadow:var(--sh-sm)`. НЕ заливать активное солид-акцентом.
- **Адаптивные поверхности**: `applySettings()` считает фактическую яркость фона каждой поверхности
  (menubar/dock/`.wdg`/`.win`/`.dialog`/`.menu`) и ставит scoped-переменные на `:root`:
  `--{mb|dock|wdg|win}-bg/-bf/-fg/-dim/-line/-accent/-good/-warn/-danger/-track`. Цвета текста/акцента
  подгоняются по контрасту через `accentAdj(hex, effL)` (WCAG). Новый элемент внутри поверхности —
  бери `var(--good/--warn/--danger/--track/--accent)` (они уже контрастно-подогнаны), НЕ хардкодь hex.
- **Профили тем**: все стилевые настройки (`THEMED_KEYS`) РАЗДЕЛЬНЫ для light/dark
  (`SET.themeProfiles`). Новую стилевую настройку — обязательно в `THEMED_KEYS`.
- Иконки — `svg(name,size)` (Lucide-подобные пути в объекте `P`/`I`). Тултипы — `tipHTML()`/`.tip`
  (тёмный пузырь, НЕ на `--ink`). Кнопки `.btn`/`.iconbtn`, сегменты `.seg`, тумблеры `.tgl`,
  диалоги `#scrim`+`.dialog`, поповеры бара `mbPop()` (класс `.mbpop`).
- Прежде чем добавлять окно/компонент — переиспользуй существующие классы/хелперы.

### 3. Язык интерфейса — ТОЛЬКО английский (рантайм-i18n СНЯТ)
UI пишется СРАЗУ по-английски — это единственный язык интерфейса. Словаря `web/i18n.js`
и рантайм-перевода БОЛЬШЕ НЕТ: их выпилили осознанно (ветка `i18n-english-only`, влита в
main — коммит «снос рантайм-i18n — english-only»). Что осталось как заглушки для
совместимости старых call-site'ов, а НЕ как живой слой перевода:
- `nasTr` в `desktop.html`/`setup.html`/`screen.html` — тождественная функция
  (`window.nasTr=function(s){return s}`), `NAS_LANG="en"` захардкожен.
- `tr()` в `nas-web.py` — identity-pass (`return text`), оставлен только чтобы
  серверные call-site'ы (Pushover, лейблы экрана) не падали.
Правила для нового кода: **никаких русских UI-строк** — пиши по-английски. Оборачивать
новую строку в `nasTr(...)` не нужно (заглушка ничего не делает; существующие обёртки не
трогай — они безвредны). Строки `meta.json` в каталоге приложений — тоже английские.
**НЕ возвращай рантайм-словарь/второй язык без явной просьбы пользователя.**

**Комментарии в коде — ПО-АНГЛИЙСКИ.** Старые русские не переводим массово (огромный diff,
шум в `git blame`), но всё новое и всё, что правишь по ходу, комментируй по-английски —
файлы переходят постепенно. Если когда-нибудь понадобится массовый перевод комментариев —
только через токенизаторы (`tokenize`+`ast` для питона; сканер состояния
строка/шаблон/regex для JS и shell), инвариант: код без комментариев обязан остаться байт
в байт прежним (наивный `sed` по `#`/`//` ломает URL, `${VAR#...}` и регулярки).

### 4. Адаптив (десктоп НЕ трогать — всё под media-queries / `isPhone()`)
Флаги: `isPhone()` (≤640), `isTablet()` (641–1024), `isTouch()` (pointer:coarse).
- Планшет+телефон (≤1024): окна **полноэкранные** (`.win{inset:40px 0 …dock 0 !important}`),
  `openWin` пропускает drag/resize; **один док внизу** — зарезервированная полоса с гориз. скроллом
  (контент резервирует его высоту, не прячется под ним); вертикальные сайдбары окон
  (`.set-nav`, `.dm2`) → горизонтальные «таблетки»/лента; тап-таргеты крупнее, инпуты 16px.
- Кнопка «назад» в шапке окна (`.wback`) → `goHome()`. `.wall-refresh` и т.п. скрыты на ≤1024.
- Не полагайся на `!important`-инсет против инлайн-стилей без проверки специфичности (был баг:
  `body.dock-left .desk{bottom:0}` перебивал мобильный отступ по специфичности).

## Кэш / доставка (важно для мобильных)
HTML отдаётся `Cache-Control: no-store` (мобильные при `no-cache` держали старую оболочку).
`web/sw.js` — network-first, навигация только из сети; при правках, если телефон видит старое —
сбросить SW (закрыть вкладку/PWA и открыть заново). При изменении shell-кэша поднимай `CACHE` (v↑).

## Ключевые грабли
- **USB-SATA мосты (UAS)** отдают диск как `ID_BUS=ata`, а не `usb` → udev-правила матчить по
  `ENV{ID_USB_DRIVER}=="?*"` (ловит usb-storage и uas), НЕ по `ID_BUS==usb`.
- **USB-диск уходит ОФЛАЙН посреди бэкапа → ext4 в аварийный read-only.** 2026-07-12 18:46:
  Ugreen (RTL9210) + Seagate ST4000LM024 (2.5" **SMR**): команда записи висела 68 с (SMR-диск
  ушёл во внутреннюю перетасовку), ядро на 30-й секунде (дефолтный таймаут SCSI) объявило
  ошибку → `Device offlined - not ready after error recovery` → `Remounting filesystem read-only`.
  Симптом в панели обманчив: прогон обрывается, а СЛЕДУЮЩИЙ красит ВСЕ папки красным — rsync не
  может даже `readdir` приёмника (`Input/output error (5)`), выглядит как «бэкап всё сломал».
  Диагноз — `dmesg` (`cmd_age=68s`, `offlined`) + `mount` (флаг `emergency_ro`). SMART при этом
  ЧИСТЫЙ (Reallocated/Pending/Uncorrect/CRC = 0, error log пуст) — носитель ни при чём, виноват
  таймаут. Лечение — udev-правило в `install_automount`: `device/timeout` = 180 с для USB-дисков.
  После такого обрыва обязательно `umount` + `e2fsck -fp` (журнал оборван).
- **Таймаута мало: UAS добивает диск сам. UAS ВЫКЛЮЧЕН ВСЕГДА с 2026-07-12** (`install_uas_off()`).
  Через 2 часа после лечения таймаутом — второй обрыв бэкапа, уже НЕ по таймауту (180 с
  применились). В 19:38 по диску пошёл `smartctl` (ATA pass-through, `opcode=0x85` — smartd
  и Scrutiny) ПОКА rsync писал; команда зависла, и `uas_eh_abort_handler` не смог отменить её
  в одиночку → `uas_eh_device_reset_handler` → reset ВСЕГО usb-устройства → оборваны все
  inflight-записи → офлайн → `emergency_ro`. Суть: **error recovery у UAS ресетит устройство
  целиком**, у usb-storage (BOT) зависшая команда падает одна. BOT «медленнее» только на бумаге:
  потолок и так гигабитная сеть (118 МБ/с) и сам HDD (~120 МБ/с).
  Одной глобальной ручки НЕТ: **`uas` вкомпилен в ядро Pi** (`modinfo uas` → `(builtin)`), поэтому
  `blacklist uas` в `modprobe.d` — пустышка; единственный рычаг `usb-storage.quirks`, и он требует
  явный VID:PID, шаблонов не понимает. Старая опция «Отключить UAS» была ловушкой: она собирала
  VID:PID только тех дисков, что воткнуты В МОМЕНТ её запуска, — Ugreen подключили позже, флаг
  горел «включено», а мост все эти месяцы жил на UAS. Поэтому опции больше НЕТ (ни в визарде, ни
  в мастере, ни в панели), а список сам себя поддерживает: udev-хук `nas-uas-off.sh` ловит любой
  мост с `ID_USB_INTERFACES ~ ":080662:"` (= умеет UAS; у флешек-BOT только `:080650:`), дописывает
  его в `/sys/module/usb_storage/parameters/quirks` И в `cmdline.txt`, затем переподнимает
  устройство (`authorized` 0→1) — оно перепривязывается к usb-storage без ребута. Живой параметр
  правится на лету, так что чинить можно и работающий бокс. Любой список «поддерживаемого железа»,
  собранный один раз, протухнет — собирай его по событию.
- **Дочерний процесс nas-web умирает при рестарте службы.** systemd убивает ВСЮ cgroup службы,
  поэтому `sudo systemctl restart nas-web` (наш обычный цикл разработки!) молча уносил sshfs-демоны
  SSH-серверов ФМ, оставляя мёртвые точки монтирования. Симптом обманчив: `mount` показывает маунт,
  а любое обращение — `Transport endpoint is not connected`, ФМ пишет «not a directory».
  Усугубляло то, что `_remote_mounted()` считал маунт живым по факту наличия в `/proc/mounts` →
  панель отвечала «уже смонтировано» и отказывалась перемонтировать. Лечение: **долгоживущий
  процесс — в СВОЙ transient-юнит** (`systemd-run --unit nas-remote-<id> --collect`, `sshfs -f`,
  `Restart=on-failure`, `ExecStopPost=umount -l`) — переживает рестарт панели, а упавший sshfs
  systemd поднимает сам. Живость маунта проверять `statvfs` (на битом FUSE ядро отвечает ENOTCONN
  сразу, без похода на сервер), а НЕ грепом `/proc/mounts`. Пароль — через `--setenv` (окружение
  юнита читает только root), НЕ в argv (`/proc` виден всем); `StandardInputText` транзиентному
  юниту задать нельзя — systemd-run его не принимает.
- **Бэкап: у задачи свой `dest`, и он не следует за общей папкой-приёмником.** Каждая задача хранит
  `dest`, посчитанный в момент добавления источника. Смена «общей папки-приёмника» меняла только
  `dest_base`, а прогон идёт по `dest` задачи → rsync молча лил в СТАРУЮ папку, пока панель
  показывала новую. Лечение: в режиме единой папки `dest` — чистая функция от (`dest_base`, `src`),
  и `nb_save()` выводит его САМ (`_nb_dest_for`, то же правило, что `destFor` в UI), не доверяя
  копии из запроса. Общее правило: если значение выводимо — выводи его на сервере, а не храни
  копию, которая разъедется с источником.
- **Роль по умолчанию в диалоге форматирования = «Данные» → флешка молча уходит в ПУЛ.**
  2026-07-12: пользователь форматировал внешний USB-диск, роль не трогал — диск стал веткой
  mergerfs (`/mnt/disk3`), попал в fstab и в пул. Вынь такой диск — часть `/mnt/storage`
  исчезнет. Лечение: у USB-диска (`tran=="usb"`) роль по умолчанию «Съёмный», а выбор «Данные»
  для съёмного диска показывает, чем это кончится. Разбор завала: строку из `/etc/fstab`,
  `umount`, `rmdir /mnt/diskN`, затем `nas-wizard.sh api mergerfs` — он пересобирает ветки из
  фактически смонтированных `/mnt/disk*` (`mounted_data_disks`), руками юнит не правь.
- **FAT32 на приёмнике бэкапа = гарантированный обрыв.** Потолок файла 4 ГиБ — rsync падает
  с `File too large (27)` и роняет ВСЮ задачу (не «пропускает файл»). Плюс vfat без `iocharset`
  монтируется как **ascii**, и любое имя с кириллицей падает с `mkdir … Invalid argument (22)`.
  Автомонтирование теперь даёт vfat `iocharset=utf8`, а панель предупреждает про 4 ГиБ ДО
  запуска. Приёмник бэкапа — только ext4 (exfat, если диск нужен и на Windows/Mac).
- **Профиль бэкапа в API передаётся ключом `p`**, а не `profile`/`id`. Ошибёшься — запрос уйдёт в
  профиль по умолчанию (последний сохранённый) и перепишет ЕГО настройки. Проверено на себе.
- **`include` в `smb.conf` ставить ТОЛЬКО в самый КОНЕЦ файла (top-level), НЕ после `[global]`.**
  `include` вклеивается текстом буквально: если он стоит внутри `[global]`, а инклуд-файл содержит
  `[global]`+секции шар, то все глобальные параметры smb.conf, идущие ПОСЛЕ строки include,
  припишутся к последней шаре инклуда (`testparm`: «Global parameter … found in service section!»)
  и потеряются из global. Проверено `testparm` 2026-07-20. Поэтому и панельный
  `include = /etc/samba/nas-shares.conf`, и `tm_ensure_include` (Time Machine — там был этот баг,
  исправлен) дописываются В КОНЕЦ, с нормализацией старого кривого размещения (снять любую строку

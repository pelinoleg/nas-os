# NAS-OS — инструкции проекта для Claude Code

Домашняя NAS-система для Raspberry Pi 5: движок настройки + веб-панель в стиле «десктоп-окружения».
Этот файл переживает переустановку (он в git), поэтому `claude`, запущенный в папке репозитория,
сразу знает архитектуру и правила. Память в `~/.claude/` — локальная и при переустановке пропадает;
источник правды по проекту — ЭТОТ файл + история git.

## Что это и где что лежит
Репозиторий: **github.com/pelinoleg/nas-os** (ветка `main`). `gh` авторизован.
- `nas-web.py` (~5000 строк) — **stdlib-only** HTTP-сервер (ThreadingHTTPServer) от **root**, порт 80.
  Отдаёт статику из `web/`, REST `/api/*`, WebSocket `/ws/term`. Мост к движку:
  `bash nas-wizard.sh api <action>` через env `NASW_*`. Никаких внешних Python-зависимостей.
- `nas-wizard.sh` (~2500 строк) — движок+инсталлятор. Интерактивный (whiptail) и headless
  API-режим (`nas-wizard.sh api <action>`, вызывается из nas-web без whiptail).
- `web/desktop.html` — весь десктоп: инлайн `<style>` + инлайн `<script>` (ванильный JS, без сборки).
- `web/setup.html` — мастер первичной настройки (отдельная страница).
- `web/i18n.js` — словарь переводов (RU→EN).
- `web/sw.js` — service worker (network-first; HTML — только из сети).
- `install.sh` — `curl -fsSL .../install.sh | sudo bash` → `git clone/reset --hard origin/main`
  в `/opt/nas-os`, ставит службу `nas-web` оттуда. **Переустановка воспроизводит текущий код 1:1.**

## Рабочий цикл разработки
1. Правишь файл.
2. **Питон**: `sudo systemctl restart nas-web` (обязательно — код в памяти службы).
   **Статика** (`web/*`): рестарт не нужен, сервер читает с диска; но браузер/кэш — см. ниже.
3. Проверка: `python3 -m py_compile nas-web.py`, `bash -n nas-wizard.sh`, для JS —
   вынуть `<script>` и `node --check`, для CSS — проверить баланс `{` / `}`.
4. **Всегда коммить И пушить** каждую законченную правку (пользователь может уйти/выключить NAS).
   Подпись коммита: `Co-Authored-By: Claude Opus 4.8 <noreply@anthropic.com>`.
5. Дев-хост = сам NAS (Pi 5). Служба `nas-web` — `enabled` (стартует после ребута сама).

## ЖЕЛЕЗНЫЕ ПРАВИЛА (нарушение = сломанная переустановка / несогласованный UI)

### 1. Полнота установщика (wizard-completeness)
Любое ГЛОБАЛЬНОЕ изменение системы (udev-правило, systemd-юнит, пакет, конфиг в `/etc`,
sudoers, fstab-логика) ОБЯЗАНО быть прописано в `nas-wizard.sh`/`install.sh`, иначе после
переустановки оно исчезнет. Если правишь «живьём» на боксе — параллельно правь генератор
в визарде. Секреты/конфиги пользователя (пароли, `nas-backup.json`, `desktop.json`, задачи)
в репо НЕ входят — они восстанавливаются из «Настройки → Бэкап настроек» (`_bk_sources()`).
**Что кладёт бэкап настроек**: весь `~/nas-config/` (тема/окна/ярлыки/обои, glance.json,
screen.json, storage.json — выбор основного хранилища, профили и история приложения Backup,
кэш иконок), `etc/nas-os/{webauth,nas-backup}.json` (пароль панели, главный NAS), `etc/nas-wizard/`
(motd, usb-import), Samba (`smb.conf` + `passdb.tdb` — пароли SMB-юзеров), `opt/stacks/*/`
(compose+.env стеков), SSH-ключ push-бэкапов (`/root/.ssh/nas-backup`). `reference/` — только
справочно (НЕ авто-восстановление): `fstab`/`snapraid.conf`/`exports` (диски другие — руками) и
`netplan/*.yaml` (Wi-Fi-пароль/статический IP; секретная секция «Сеть»). Секретные секции
(webauth/nasbackup/network) прячутся за отдельным подтверждением при восстановлении. Дефолтный
путь — основное хранилище (пул или выбранный USB через `storage_root()`), без него — системный
`/var/backups/nas-os` (эфемерно, панель предупреждает).

### 2. Дизайн-система (весь новый UI обязан следовать)
Токены — CSS-переменные, тема через `:root[data-theme]` + `@media prefers-color-scheme`.
- НЕ использовать солид-цвета (`#hex`, `var(--bg)`, `var(--surface)`) как заливку карточек/панелей
  ВНУТРИ окон — только стекло `--glass`/`--card`. `var(--bg)` (сплошной) = ТОЛЬКО рабочий стол/body.
- **Elevation в тёмной теме = белый слой с прозрачностью** поверх материала (не осветление того же цвета).
- **Активные табы/сегменты/сайдбары — «белая фишка»**: `background:var(--sel)` + акцентный ТЕКСТ +
  `box-shadow:var(--sh-sm)`. НЕ заливать активное солид-акцентом.
- **Адаптивные поверхности**: `applySettings()` считает фактическую яркость фона каждой поверхности
  (menubar/dock/`.wdg`/`.win`/`.dialog`/`.menu`) и ставит scoped-переменные на `:root`:
  `--{mb|dock|wdg|win}-bg/-bf/-fg/-dim/-line/-accent/-good/-warn/-danger/-track`. Цвета текста/акцента
  подгоняются по контрасту через `accentAdj(hex, effL)` (WCAG). Новый элемент внутри поверхности —
  бери `var(--good/--warn/--danger/--track/--accent)` (они уже контрастно-подогнаны), НЕ хардкодь hex.
- **Профили тем**: все стилевые настройки (`THEMED_KEYS`) РАЗДЕЛЬНЫ для light/dark
  (`SET.themeProfiles`). Новую стилевую настройку — обязательно в `THEMED_KEYS`.
- Иконки — `svg(name,size)` (Lucide-подобные пути в объекте `P`/`I`). Тултипы — `tipHTML()`/`.tip`
  (тёмный пузырь, НЕ на `--ink`). Кнопки `.btn`/`.iconbtn`, сегменты `.seg`, тумблеры `.tgl`,
  диалоги `#scrim`+`.dialog`, поповеры бара `mbPop()` (класс `.mbpop`).
- Прежде чем добавлять окно/компонент — переиспользуй существующие классы/хелперы.

### 3. i18n (EN — язык по умолчанию, RU — исходный)
**UI-строки** пишутся по-русски — это исходный язык для `i18n.js`. **Любая новая строка UI
СРАЗУ получает перевод в `web/i18n.js` (`NAS_I18N.en`)**. Ключ = точный русский фрагмент
(по «словам», границы — не-кириллица; длинные фразы приоритетнее). После правок прогоняй
детектор непокрытых кириллических «слов».
**Перевод цепляется ТОЛЬКО за сеттер `innerHTML`** (в `textContent` движок не лезет намеренно:
им пишут xterm и CodeMirror — перевод поехал бы по выводу терминала). Значит кириллица,
присвоенная в `.textContent`/`.value`/`.title`, остаётся русской в EN, даже когда ключ в словаре
есть. Такие места — руками через `nasTr(...)`. Ловится грепом: `textContent=` рядом с кириллицей
и без `nasTr` (2026-07-13: так «Тест… (~15 c)» и «меряю…» в виджете сети жили непереведёнными
при живом ключе в `i18n.js`).

**Комментарии в коде — ПО-АНГЛИЙСКИ.** Старые русские не переводим массово (огромный diff,
шум в `git blame`), но всё новое и всё, что правишь по ходу, комментируй по-английски —
файлы переходят постепенно. Комментарии ≠ строки: русские строки остаются.
Если когда-нибудь всё же понадобится массовый перевод — только через токенизаторы
(`tokenize`+`ast` для питона; сканер состояния строка/шаблон/regex для JS и shell) и с
проверкой инварианта: код без комментариев обязан остаться байт в байт прежним.
Наивный `sed` по `#`/`//` ломает URL, `${VAR#...}` и регулярки.

### 4. Адаптив (десктоп НЕ трогать — всё под media-queries / `isPhone()`)
Флаги: `isPhone()` (≤640), `isTablet()` (641–1024), `isTouch()` (pointer:coarse).
- Планшет+телефон (≤1024): окна **полноэкранные** (`.win{inset:40px 0 …dock 0 !important}`),
  `openWin` пропускает drag/resize; **один док внизу** — зарезервированная полоса с гориз. скроллом
  (контент резервирует его высоту, не прячется под ним); вертикальные сайдбары окон
  (`.set-nav`, `.dm2`) → горизонтальные «таблетки»/лента; тап-таргеты крупнее, инпуты 16px.
- Кнопка «назад» в шапке окна (`.wback`) → `goHome()`. `.wall-refresh` и т.п. скрыты на ≤1024.
- Не полагайся на `!important`-инсет против инлайн-стилей без проверки специфичности (был баг:
  `body.dock-left .desk{bottom:0}` перебивал мобильный отступ по специфичности).

## Кэш / доставка (важно для мобильных)
HTML отдаётся `Cache-Control: no-store` (мобильные при `no-cache` держали старую оболочку).
`web/sw.js` — network-first, навигация только из сети; при правках, если телефон видит старое —
сбросить SW (закрыть вкладку/PWA и открыть заново). При изменении shell-кэша поднимай `CACHE` (v↑).

## Ключевые грабли
- **USB-SATA мосты (UAS)** отдают диск как `ID_BUS=ata`, а не `usb` → udev-правила матчить по
  `ENV{ID_USB_DRIVER}=="?*"` (ловит usb-storage и uas), НЕ по `ID_BUS==usb`.
- **USB-диск уходит ОФЛАЙН посреди бэкапа → ext4 в аварийный read-only.** 2026-07-12 18:46:
  Ugreen (RTL9210) + Seagate ST4000LM024 (2.5" **SMR**): команда записи висела 68 с (SMR-диск
  ушёл во внутреннюю перетасовку), ядро на 30-й секунде (дефолтный таймаут SCSI) объявило
  ошибку → `Device offlined - not ready after error recovery` → `Remounting filesystem read-only`.
  Симптом в панели обманчив: прогон обрывается, а СЛЕДУЮЩИЙ красит ВСЕ папки красным — rsync не
  может даже `readdir` приёмника (`Input/output error (5)`), выглядит как «бэкап всё сломал».
  Диагноз — `dmesg` (`cmd_age=68s`, `offlined`) + `mount` (флаг `emergency_ro`). SMART при этом
  ЧИСТЫЙ (Reallocated/Pending/Uncorrect/CRC = 0, error log пуст) — носитель ни при чём, виноват
  таймаут. Лечение — udev-правило в `install_automount`: `device/timeout` = 180 с для USB-дисков.
  После такого обрыва обязательно `umount` + `e2fsck -fp` (журнал оборван).
- **Таймаута мало: UAS добивает диск сам. UAS ВЫКЛЮЧЕН ВСЕГДА с 2026-07-12** (`install_uas_off()`).
  Через 2 часа после лечения таймаутом — второй обрыв бэкапа, уже НЕ по таймауту (180 с
  применились). В 19:38 по диску пошёл `smartctl` (ATA pass-through, `opcode=0x85` — smartd
  и Scrutiny) ПОКА rsync писал; команда зависла, и `uas_eh_abort_handler` не смог отменить её
  в одиночку → `uas_eh_device_reset_handler` → reset ВСЕГО usb-устройства → оборваны все
  inflight-записи → офлайн → `emergency_ro`. Суть: **error recovery у UAS ресетит устройство
  целиком**, у usb-storage (BOT) зависшая команда падает одна. BOT «медленнее» только на бумаге:
  потолок и так гигабитная сеть (118 МБ/с) и сам HDD (~120 МБ/с).
  Одной глобальной ручки НЕТ: **`uas` вкомпилен в ядро Pi** (`modinfo uas` → `(builtin)`), поэтому
  `blacklist uas` в `modprobe.d` — пустышка; единственный рычаг `usb-storage.quirks`, и он требует
  явный VID:PID, шаблонов не понимает. Старая опция «Отключить UAS» была ловушкой: она собирала
  VID:PID только тех дисков, что воткнуты В МОМЕНТ её запуска, — Ugreen подключили позже, флаг
  горел «включено», а мост все эти месяцы жил на UAS. Поэтому опции больше НЕТ (ни в визарде, ни
  в мастере, ни в панели), а список сам себя поддерживает: udev-хук `nas-uas-off.sh` ловит любой
  мост с `ID_USB_INTERFACES ~ ":080662:"` (= умеет UAS; у флешек-BOT только `:080650:`), дописывает
  его в `/sys/module/usb_storage/parameters/quirks` И в `cmdline.txt`, затем переподнимает
  устройство (`authorized` 0→1) — оно перепривязывается к usb-storage без ребута. Живой параметр
  правится на лету, так что чинить можно и работающий бокс. Любой список «поддерживаемого железа»,
  собранный один раз, протухнет — собирай его по событию.
- **Дочерний процесс nas-web умирает при рестарте службы.** systemd убивает ВСЮ cgroup службы,
  поэтому `sudo systemctl restart nas-web` (наш обычный цикл разработки!) молча уносил sshfs-демоны
  SSH-серверов ФМ, оставляя мёртвые точки монтирования. Симптом обманчив: `mount` показывает маунт,
  а любое обращение — `Transport endpoint is not connected`, ФМ пишет «not a directory».
  Усугубляло то, что `_remote_mounted()` считал маунт живым по факту наличия в `/proc/mounts` →
  панель отвечала «уже смонтировано» и отказывалась перемонтировать. Лечение: **долгоживущий
  процесс — в СВОЙ transient-юнит** (`systemd-run --unit nas-remote-<id> --collect`, `sshfs -f`,
  `Restart=on-failure`, `ExecStopPost=umount -l`) — переживает рестарт панели, а упавший sshfs
  systemd поднимает сам. Живость маунта проверять `statvfs` (на битом FUSE ядро отвечает ENOTCONN
  сразу, без похода на сервер), а НЕ грепом `/proc/mounts`. Пароль — через `--setenv` (окружение
  юнита читает только root), НЕ в argv (`/proc` виден всем); `StandardInputText` транзиентному
  юниту задать нельзя — systemd-run его не принимает.
- **Бэкап: у задачи свой `dest`, и он не следует за общей папкой-приёмником.** Каждая задача хранит
  `dest`, посчитанный в момент добавления источника. Смена «общей папки-приёмника» меняла только
  `dest_base`, а прогон идёт по `dest` задачи → rsync молча лил в СТАРУЮ папку, пока панель
  показывала новую. Лечение: в режиме единой папки `dest` — чистая функция от (`dest_base`, `src`),
  и `nb_save()` выводит его САМ (`_nb_dest_for`, то же правило, что `destFor` в UI), не доверяя
  копии из запроса. Общее правило: если значение выводимо — выводи его на сервере, а не храни
  копию, которая разъедется с источником.
- **Роль по умолчанию в диалоге форматирования = «Данные» → флешка молча уходит в ПУЛ.**
  2026-07-12: пользователь форматировал внешний USB-диск, роль не трогал — диск стал веткой
  mergerfs (`/mnt/disk3`), попал в fstab и в пул. Вынь такой диск — часть `/mnt/storage`
  исчезнет. Лечение: у USB-диска (`tran=="usb"`) роль по умолчанию «Съёмный», а выбор «Данные»
  для съёмного диска показывает, чем это кончится. Разбор завала: строку из `/etc/fstab`,
  `umount`, `rmdir /mnt/diskN`, затем `nas-wizard.sh api mergerfs` — он пересобирает ветки из
  фактически смонтированных `/mnt/disk*` (`mounted_data_disks`), руками юнит не правь.
- **FAT32 на приёмнике бэкапа = гарантированный обрыв.** Потолок файла 4 ГиБ — rsync падает
  с `File too large (27)` и роняет ВСЮ задачу (не «пропускает файл»). Плюс vfat без `iocharset`
  монтируется как **ascii**, и любое имя с кириллицей падает с `mkdir … Invalid argument (22)`.
  Автомонтирование теперь даёт vfat `iocharset=utf8`, а панель предупреждает про 4 ГиБ ДО
  запуска. Приёмник бэкапа — только ext4 (exfat, если диск нужен и на Windows/Mac).
- **Профиль бэкапа в API передаётся ключом `p`**, а не `profile`/`id`. Ошибёшься — запрос уйдёт в
  профиль по умолчанию (последний сохранённый) и перепишет ЕГО настройки. Проверено на себе.
- `getComputedStyle().getPropertyValue('--x')` НЕ резолвит `var()` внутри кастом-проперти —
  читай уже-вычисленные `--{surface}-good/…` с `:root` (они хексы, поставлены `setProperty`).
- `.menu button{width:100%}` растягивает любую кнопку в поповере — для компактных нужен оверрайд.
- Уведомления фоновых агентов могут врать — сверяй файлы на диске.
- **БП 15 Вт (3 А) + 3×NVMe + USB-диск = PMIC вырубает плату под нагрузкой.** Реальный случай
  2026-07-11 18:32: во время Time Machine-бэкапа 5V просело, PMIC защитно выключил Pi — журнал
  обрывается без shutdown, холодный старт (`rsts 0`), включение только кнопкой. Watchdog тут
  бессилен (питания нет). Диагноз: `/proc/device-tree/chosen/power/max_current` = 3000 (нужно
  5000 = официальный 27 Вт БП; слабый кабель тоже роняет переговоры PD до 3 А).
  `usb_max_current_enable=1` при слабом БП усугубляет. Софт-защита от зависаний панели —
  `web_selfheal()` в `nas-netguard.sh` (проба HTTP на localhost, 3 промаха → SIGUSR1-дамп
  стеков в журнал → рестарт nas-web). **РЕШЕНО 2026-07-11: БП заменён на официальный 27 Вт,
  переговоры PD дают max_current = 5000, `throttled=0x0`.** Значение видно в виджете
  «Система» (строка «Питание», `stats().psu_ma`); < 5000 → жёлтое «нужно 5 A» = снова
  слабый БП или кабель.
- **comitup ворует wlan0 навсегда. ВЫКЛЮЧЕН 2026-07-12** (`disable_comitup()` в визарде: disable+mask
  всех юнитов `comitup*` + удаление профилей `comitup-*`). Пока он жил: NAS на проводе → netguard
  держит autoconnect домашнего Wi-Fi выключенным → comitup читает «у wlan0 нет сети» и поднимает
  хотспот `comitup-*` по 10–29 раз в час; а при реальном обрыве провода wlan0 возвращался В ХОТСПОТ,
  а не в домашнюю сеть, и давал каждой попытке возврата ~20 с — мало для скана 5 ГГц → NAS жив, но
  недоступен (2026-07-11, 03:00–09:09). Резерв Wi-Fi comitup не нужен: домашний профиль лежит в NM,
  netguard поднимает его сам. `wifi_rescue()` в `nas-netguard.sh` оставлен на случай чужого образа.
- **Ежечасный спидтест = ложный обрыв провода.** MySpeed по умолчанию `cron: "0 * * * *"` — ровно в
  :00 он выбирал весь канал (~300/200 Мбит/с, 20–30 с), роутер под нагрузкой ронял ICMP/ARP, и
  netguard объявлял провод мёртвым. Отсюда правило в `eth_healthy()`: жёсткий обрыв (нет carrier /
  адреса / маршрута) — failover сразу, а МОЛЧАНИЕ ШЛЮЗА — только после `GWMISS_MAX`=3 промахов
  подряд. Любую новую проверку «жив ли линк» делать так же: под нагрузкой сеть врёт.
- **Спидтест роняет НЕ NAS, а промежуточный роутер (GL.iNet Beryl).** 2026-07-12 19:31, повтор уже
  ПОСЛЕ фикса `GWMISS_MAX`: спидтест → на 18 с отвалились SSH, панель и ping шлюза, сам MySpeed упал
  с `ConfigurationError`. NAS чист: carrier eth0 не падал, линк 1000 Мбит/с, conntrack 309/262144,
  nas-web не перезапускался, netguard failover НЕ делал (защита сработала — `шлюз не ответил (1/3)`,
  провод удержан). Топология (её не видно, пока не посмотришь): `ip neigh` — **ВСЕ** адреса дома
  резолвятся в один MAC `94:83:c4:…` (GL Technologies), а TTL-зонд (`ping -t 1/2/3`) даёт
  `192.168.8.1` (Beryl) → `192.168.1.1` (Flint 2) → CGNAT. То есть NAS сидит на «острове» за Beryl,
  тот раздаёт proxy-ARP и МАРШРУТИЗИРУЕТ (мост не уменьшал бы TTL) — в таком режиме аппаратный
  оффлоад отключается, и весь трафик NAS↔дом молотит CPU дорожного роутера. 300/200 Мбит/с в
  параллельных потоках забивают его в полку → он перестаёт форвардить ВСЁ. Тот же эффект даст
  большая заливка на NAS или Time Machine. Правильное лечение — физическое: воткнуть Pi напрямую
  в Flint 2. Диагностика на будущее: `ping -t N` + `ip neigh | grep <mac>` показывают скрытый хоп
  за минуту; пока не посмотришь — «падает NAS» выглядит правдоподобно, а падает не он.
- **eth0 + wlan0 в одной подсети = катастрофа.** NM снимает on-link маршрут подсети по ложному
  ACD-конфликту → ответы соседям уходят через шлюз → путь асимметричный → роутер впрыскивает RST.
  Симптом обманчив: `ping` идёт, IPv6/SSH живы, а HTTP висит на «loading…». Лечит `nas-netguard.sh`
  («один активный линк за раз», eth0 главный). Не «чинить маршрут» — держать один интерфейс.
- **`nmcli device disconnect` НЕ держит Wi-Fi выключенным.** У профиля `autoconnect=yes`, NM поднимает
  его снова через пару минут → сторож гасит → NM поднимает (реально: 63 «отключаю wlan0» за 12 ч).
  Каждый всплеск = второй адрес в той же подсети на секунды = обманчивый обрыв HTTP (см. выше);
  клиент по `pi5.local` при этом может уехать в стухший mDNS-адрес и «терять» NAS на минуты.
  Лечение — `wifi_autoconnect no|yes` в netguard: пока провод жив, autoconnect домашнего Wi-Fi
  выключен совсем; провод отвалился → возвращаем `yes` и поднимаем (фолбэк не теряется: таймер
  15 с + хук NM-диспетчера на carrier-down).
- **`${VAR:-{a}-{b}}` в bash ломается**: первая `}` закрывает подстановку, хвост клеится текстом.
  Дефолты с фигурными скобками выносить отдельной строкой. Проверять `${VAR+set}`, а не `:-`,
  если пустое значение — осознанный режим (напр. «без подпапки»).
- `printf '%-12s'` меряет БАЙТЫ: кириллица разъезжает по колонкам. Вывод в терминал — латиницей.
- В `pam_motd`/udev-хуках нельзя `smartctl`/`hdparm` (будят спящие диски) и синхронный `nmcli`
  из NM-dispatcher (дедлок: NM ждёт скрипт, скрипт ждёт NM → `systemctl start --no-block`).
- **Частый опрос × дорогой источник = бокс в полке.** 2026-07-13: экран опрашивал `/api/screen/data`
  раз в 1.5 с, а payload честно пересчитывал ВСЕ плитки Glance — среди них `updates` (запускает
  `apt-get -s upgrade`) и `disktemp` (ходит к дискам). Получилось до 40 запусков apt в минуту:
  load 14, CPU 100 %, киоску не доставалось процессора и экран стоял ПУСТОЙ (выглядело как баг
  вёрстки, а не как перегрузка). Лечение — свой TTL у каждого источника (`SCREEN_TILE_TTL`:
  updates 5 мин, disktemp/snapraid 2 мин, inet 1 мин, docker 15 с, дешёвые 2 с; тяжёлое со второй
  страницы — 30 с и только когда она открыта). ПРАВИЛО: ускоряя опрос, сначала посмотри, во что
  обходится КАЖДОЕ поле ответа — частота умножается на цену.
- **Курсор мыши на настенном экране: указателей больше, чем кажется.** Гасили по одному и трижды
  промахивались. На Pi 4 указателем для libinput оказываются: мышь USB-донгла (`ID_INPUT_MOUSE`),
  его же `Consumer Control` (есть относительные оси `REL=1040`, но метки `ID_INPUT_MOUSE` НЕТ —
  правило по мыши его не ловит) и даже джеки HDMI (`ID_INPUT_POINTINGSTICK`). Пока жив хоть один,
  композитор рисует курсор — `cursor:none` в CSS, `XCURSOR_SIZE` и пустая тема курсора не помогают.
  Лечение — перевернуть правило: `SUBSYSTEM=="input", ENV{ID_INPUT}=="1",
  ENV{ID_INPUT_TOUCHSCREEN}!="1", ENV{LIBINPUT_IGNORE_DEVICE}="1"` (киоску — только палец).
  Проверять по открытым дескрипторам: `ls -l /proc/$(pgrep -x cage)/fd | grep event`.
- **`_run()` возвращает СЛОВАРЬ `{ok,code,log}`, а не строку.** `.strip()` у него → AttributeError →
  ручка отвечает 500 → `api()` в панели бросает → вкладка настроек рисует только заголовок.
  Симптом обманчив: выглядит как «UI не обновился», а это упавший запрос.
- **Параметр, обработанный только в одной ветке роутера, «работает» и не работает одновременно.**
  `?screen=1` (ужатые обои) я обработал в loopback-ветке — киоск получал 68 КБ, а браузер из
  локалки шёл в авторизованную ветку и получал оригинал 1.6 МБ. Проверять надо ТЕМ ЖЕ путём,
  каким ходит пользователь: `curl` с `127.0.0.1` доказывал «всё хорошо», пока пользователь видел
  обратное.
- **Скриншот с `grim -c` рисует курсор сам.** Флаг просит композитор дорисовать указатель — по
  такому кадру нельзя судить, виден ли курсор на панели. Для проверки вёрстки — `grim` без `-c`.
- **Прогресс — НЕ заполнение: порогами его красить нельзя.** `bar(pct)` на экране по умолчанию
  красит полосу как заливку диска (≥75 % жёлтый, ≥90 % красный). Той же функцией рисовали ход
  бэкапа и USB-импорта → успешный прогон под конец становился КРАСНЫМ, и это читалось как авария.
  Прогресс-полоса — свой класс (`.bar>i.pg`, акцент). Правило шире одной функции: у «сколько
  осталось» и «сколько занято» ПРОТИВОПОЛОЖНЫЙ знак — 95 % копии это хорошо, 95 % диска плохо.
- **История на настенном экране обязана пережить ребут.** Плашки USB-импорта/бэкапов должны
  отвечать «когда это было в последний раз» и с холодного старта, а прогресс-файлы импорта живут
  в `/run` (умирают с боксом) и вычищаются через 600 с после финиша. Постоянный след — тот, что
  `usb_ops_sync()` пишет в `ops-history.json`; экран читает его (`usb_import_history()`), а не
  `/run`. Слова «готово» на плашке нет намеренно: зелёная точка + ДАТА («вчера 21:40») отвечают
  и «всё хорошо», и «когда» — а «готово» без даты не отвечает ни на что.
- **JS на боксе без node проверять нечем.** Дважды ловил синтаксическую ошибку только по пустой
  странице (скрипт не выполняется — значит и `catch` не сработает, и ошибки не увидишь). На боксе
  с экраном ставь `nodejs` и гоняй `node --check` на вынутом `<script>` — это дешевле, чем
  отлаживать белый экран.
- **Киоск-браузер: флаги Chromium НЕ убирают всплывашки — только managed policy.** 2026-07-13,
  локальный экран: `--disable-features=Translate,TranslateUI` не убрал плашку «Перевести страницу»
  (Chromium 150), она честно висела поверх дашборда. Работает только
  `/etc/chromium/policies/managed/*.json` (`TranslateEnabled:false`, `PasswordManagerEnabled:false`,
  `DefaultNotificationsSetting:2` …) — её кладёт `install_screen()`. Профиль при этом надо снести:
  показ плашки запоминается в нём. Второе: `cage` берёт VT, поэтому юнит обязан
  `Conflicts=getty@tty1` + `PAMName=login` (без logind-сессии нет seat0 → нет DRM master).
- **Обои под стеклом: `z-index:-1` уезжает ЗА фон `body`.** Фон-слой с отрицательным z-index
  рисуется ниже фона body, и непрозрачный `body{background:#0a0f18}` его просто закрашивал —
  «обои не грузятся» при живом `/api/wallpaper/img` (200, webp). Фон держать на `html`, `body`
  оставлять прозрачным.
- **Бокс без дисков = красный `smartmontools.service`.** 2026-07-13, установка на Pi 4 (грузится
  с SD-карты, NVMe-слота нет вовсе, USB-диски ещё не воткнуты): `smartmontools` лежит в
  `STACK_PACKAGES`, Debian включает юнит сам, `DEVICESCAN` не находит НИ ОДНОГО устройства →
  smartd выходит с кодом 17 («No devices to monitor») → `failed`, и панель показывает упавшую
  службу. Мониторить и правда нечего — это не поломка, а ложная тревога. Лечит
  `install_smartd_guard()` (вызывается там же, где ставится NAS-стек): `ConditionPathExistsGlob`
  `|/dev/sd*` + `|/dev/nvme*n*` (дисков нет → юнит не стартует, `inactive`, а не `failed`);
  `Type=simple` + `SuccessExitStatus=17` для случая «диск есть, но SMART не отдаёт» (флешка) —
  ОДНОГО `SuccessExitStatus` мало: при штатном `Type=notify` выход ДО `READY=1` даёт
  `Result=protocol` → `failed` в любом случае; udev `99-nas-smartd.rules` поднимает smartd, когда
  появился первый настоящий диск (**только `start` и только если не запущен** — слепой `restart`
  на каждый `add` переопрашивал бы ВСЕ диски и будил спящие).
- **Pi без RTC: первые секунды после загрузки часы СТУХШИЕ (вчерашние), и любой писатель
  журналов по epoch их отравляет.** 2026-07-14, pi4: netguard успел отработать ДО синка NTP —
  перезаписал `avail.beat` временем из прошлого (стёр реальную метку «последний раз жив»), а
  первый прогон после синка написал «off» задним числом от этого мусора: начало ночного простоя
  уехало с ~22:00 на 13:34, монотонный клэмп читателя проглотил весь вчерашний «up», и полоса
  24 ч стала целиком красной. Диагноз обманчив дважды: `journalctl --list-boots` показывает ОДНУ
  загрузку «с 13:33 вчера» (первые записи новой загрузки идут со стухшим штампом, потом время
  прыгает на +18 ч), а `systemctl status` таймера — «active 18h». Правду говорит `/proc/uptime`.
  Лечение в `avail_track`: `now < beat - 60` → часы неправильные, в журнал не пишем вовсе (60 с —
  допуск на мелкие шаги NTP назад). Правило шире: сравнивать «сейчас» с сохранённой меткой можно
  только убедившись, что «сейчас» ≥ метки — на боксе без RTC время после ребута идёт НЕ вперёд.
- **Правка `nas-wizard.sh` НЕ обновляет установленные им скрипты.** Вчерашний фикс avail-журнала
  пролежал в репо, а `/usr/local/bin/nas-netguard.sh` остался старым — данные продолжали
  портиться «починенным» кодом. После правки генератора — сразу `nas-wizard.sh api <секция>`
  (для сторожа: `api netguard`) на КАЖДОМ боксе, где это уже стоит.

## ⚠️ pi5 МЁРТВ (2026-07-14)
Основной бокс (Pi 5, 3×NVMe + USB-диск) умер ВО ВРЕМЯ БЭКАПА и больше не включается.
Дев-хост и рабочий NAS временно — **pi4** (бокс с тач-экраном, SD-карта, дисков пока нет).
История питания pi5 в граблях: PMIC уже вырубал плату под нагрузкой при слабом БП
(вылечено 27 Вт БП 2026-07-11) — смерть под нагрузкой бэкапа укладывается в тот же паттерн.
При реанимации сначала: снять ВСЁ (NVMe-шляпа, USB-диски), холодный сброс PMIC
(питание вынуть на 5+ мин), проверить БП/кабель на pi4 (у обоих USB-C). Приёмник бэкапа
после обрыва — обязательно `e2fsck -fp` (журнал оборван, см. граблю про emergency_ro).
Данные целы независимо от платы: ветки пула — обычные ext4, mergerfs хранит файлы
целиком по веткам → любой NVMe читается через USB-адаптер на pi4.

запуски» (бэкапы по расписанию + cron) и «Здоровье дисков» (SMART-вердикт, температура,
наработка). Часы при простое (`screen.json:clock_min`, крупное время + основные показатели,
дрейф от выгорания; тап убирает). Действия: запуск бэкапа и безопасное извлечение USB
(двойной тап «Извлечь» → «Точно извлечь?», взвод живёт в JS-переменной — лист перерисовывается
каждый опрос). Гашение подсветки — `bl_power=4` (brightness=0 у этой панели НЕ гасит подсветку
до конца). Пробуждение: пока `dark:true`, клиент держит невидимый щит — первый тап только будит,
не нажимая плашку под пальцем (грейс 4 с от стухшего ответа опроса). Страница масштабируемая:
логическая сцена 800×480, `fit()` скейлит body под любой вьюпорт (телефон/планшет/другая панель),
раскладка не меняется; не-loopback получает класс `.remote` (курсор возвращается). `?page=N` —
открыть сразу N-ю страницу (нужно для скриншотов: свайп в headless не сделать). Кнопки
сна и питания в шапке (питание → две плитки на пол-экрана). Рендерер — `cage` + `chromium --kiosk`
(`nas-screen.service`, tty1), ставит `install_screen()` в визарде ТОЛЬКО если панель реально
подключена (иначе headless-бокс тянул бы 600 МБ хрома). `/screen` и `/api/screen/*` — до auth-гейта,
но ТОЛЬКО с loopback (`_local()`): на экране, до которого дотягиваешься рукой, пароль не спрашиваем,
из локалки те же пути требуют логина. Яркость/ночной режим/интервал опроса — `screen.json` +
`_screen_tick()` в monitor_loop (тревога `health=bad` зажигает экран даже ночью; касание будит
через `POST /api/screen/act {a:"touch"}`). Настройки → вкладка «Тачскрин».
**Заметки** (`winNotes` + модуль notes в nas-web.py): обычные .md с frontmatter в
`notes_root()` (дефолт `/mnt/storage/notes`, конфиг `~/nas-config/notes.json`, миграция
`/api/notes/root`); `_assets/` для картинок, `.trash/` для удалённых. Редактор — vendored
Toast UI (web/tui-editor*.js/css, MIT, офлайн — НЕ обновлять с CDN бездумно) + плагин
подсветки кода (web/tui-hl.*). Клик по картинке — меню размеров (ширина хранится тегом
`<img width>` в markdown), дабл-клик — лайтбокс. Таб настроек «Заметки» (`notesTab`).
Сохранение — оптимистическая блокировка по mtime (`base_mtime`; конфликт → диалог
перезаписать/открыть свежую; unload-флаш через `fetch keepalive` с `conflict_copy` —
сервер паркует текст соседним файлом «(конфликт …)»). Корзина хранит происхождение
(`.trash/<день>/.origins.json`) — «Вернуть» кладёт на исходное место. `notes_gc()` в
`maintenance_daily`: срок корзины `maintenance.json:notes_trash_days`, сиротская
`.history` (заметки нет нигде), несвязанные `_assets` (имени нет ни в одном .md,
файл старше 7 дней).
**Приложения из каталога** (2026-07-16: отдельная вкладка-магазин УБРАНА — путала и
дублировала управление стеками в overview). Каталог = `services/<id>/` в репо (compose +
`meta.json`: name/desc/category/icon/port/fields; строки meta — русские, переводы руками в
i18n.js, детектор json не сканирует; `stack_catalog()` → `/api/store`). Рецепты, которых
ещё НЕТ в `/opt/stacks`, показываются **в сайдбаре окна Docker секцией «Available»**
(приглушённые); клик → пейн `dkAvailPane` в детали (поля + «Install & start»), установка =
копия папки → `/opt/stacks/<id>/compose.yaml` + `.env` из полей (`{storage}/{tz}/{host}/{rand}`,
merge с сущест-м) → стрим `up` через `/api/stack/stream`; по завершении ВСЕГДА переходим на
готовый стек (никаких залипших модалок — старый баг). `stack_install()`. В пейне есть
раскрывашка «Show / edit compose & .env» (`GET /api/store/recipe` → сырой compose рецепта +
.env-превью с подстановками; ленивая загрузка): правки в textarea'ах переопределяют
копию рецепта и .env из полей (`stack_install(sid, values, compose, env)`). Рецепт в
`services/` — только чтение (git); удаление стека (`stack_delete`) сносит `/opt/stacks/<id>`
целиком, но volume с данными остаются (`down` без `-v`), и приложение снова видно в «Available». Установленные —
обычные стеки. **Свои карточки на стол УБРАНЫ** (`store.json:custom` больше не читается,
ярлыки идут только из label'ов `web-desktop.*`). Хост в `web-desktop.url` панель
переписывает на реальный LAN-адрес бокса (`_app_host_url`) — рецепты host-agnostic
(`localhost`-плейсхолдер), старый хардкод `pi5.local` больше не ведёт в никуда.
**Реплика** (рецепт `meta.json:replica`, пока Immich) — ОСТАЛАСЬ, кнопка «Replica…» в детали
установленного стека (`stReplicaDlg`): SSH-дамп Postgres с источника + rsync медиатеки
(`/api/store/replica/run`, дамп в `~/nas-config/replica/<id>/`), восстановление поднимает
стек ТОЙ ЖЕ версией, что источник (тег из `docker inspect` при синке → `IMMICH_VERSION` в
.env). `store.json` = только replica-конфиги (пароль SSH → секретная секция бэкапа настроек).
Реплика: `auto="HH:MM"` в конфиге — ежедневный автосинк (`_replica_tick` в
monitor_loop, лог в `~/nas-config/replica/<id>/sync.log`, провал → уведомление).
**SSH-серверы в ФМ**: секция «Серверы» в сайдбаре — sshfs-маунты в `/mnt/remote/<id>`
(`~/nas-config/remotes.json` с паролями → секретная секция бэкапа; opts
reconnect+ServerAlive, allow_other; пакет sshfs в UTIL_PACKAGES визарда).
Смонтированный сервер = обычная папка, все операции ФМ работают как есть.
